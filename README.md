# ⚖️ Legal Case Management Intelligence Platform

> An end-to-end analytics platform for a plaintiff-side / mass-tort law firm —
> tracking client intake, case workup, case quality, case-manager productivity,
> forecasting, and operational bottlenecks, from raw data to executive decision
> support.

**Role context:** Built as a portfolio project for a **Strategic Operations
Business Analyst** position. It demonstrates the full analyst stack — data
modeling, advanced SQL, Python analytics & machine learning, BI dashboard
design, and business-analysis documentation — applied to a realistic legal
operations problem.

---

## 📌 Project Overview

Plaintiff-side firms live and die by their **litigation pipeline**: how quickly
intake converts to a signed client, how fast medical records and documents are
collected, how reliably matters are "worked up" to a demand, and how
productively case managers move that inventory. This project simulates a
mid-size firm's operational data and builds the analytics layer leadership needs
to manage that pipeline.

The platform answers questions like:
- Where do matters stall between intake and resolution?
- Which case types and channels convert to completed workup — and which lag?
- How productive is each case manager, adjusted for caseload?
- How many workups will we complete next quarter (capacity planning)?
- Which open matters are **at risk** of never completing workup, so we can
  intervene today?
- Can we trust our data?

---

## 🧨 Business Problem

The firm's leadership operates largely on intuition and lagging spreadsheets:

- **No pipeline visibility** — partners cannot see where the ~2,180 open matters
  sit in the workup funnel or where they pile up.
- **Records bottleneck** — medical-record and document collection is the slowest,
  least-visible stage, silently extending cycle time.
- **Mass-tort underperformance** — high-volume mass-tort dockets complete workup
  far less often than core PI work, but nobody has quantified the gap.
- **Productivity blind spots** — case-manager performance is judged by raw counts,
  not caseload-adjusted completion.
- **Reactive, not predictive** — the firm finds out a matter stalled *after* it
  has aged out, instead of flagging risk early.
- **Data trust** — duplicates, missing fields and impossible values quietly
  corrupt every report.

This platform replaces intuition with an **operational decision-support system**.

## 🗃️ Dataset Description

Synthetic but **internally consistent and referentially intact** data for a firm
with a 24-month operating history (**2023-07 → 2025-06**, snapshot date
**2025-06-30**). Generated by [`python/generate_data.py`](python/generate_data.py)
with a fixed seed for reproducibility.

| Table | Grain | Rows | Notes |
|-------|-------|-----:|-------|
| `clients` | one per client | 2,540 | de-duplicated to **2,500** in cleaning |
| `attorneys` | one per attorney | 12 | dimension |
| `case_managers` | one per manager | 18 | dimension (capacity, team) |
| `cases` | **one per matter** | 2,743 | core fact + operational metrics |
| `activities` | one per touchpoint | ~17.4k | time tracking |
| `documents` | one per document | ~15.1k | received / missing / pending |
| `medical_records` | one per record | ~5.9k | received / outstanding |
| `settlements` | one per settled case | 329 | economics (fee, lien, net) |
| `call_logs` | one per call | ~15.0k | inbound / outbound |

**Workup funnel stages:** Intake → Retainer Signed → Records Requested → Records
Received → Workup Complete → Demand Sent → Negotiation → Resolved.

**Deliberately injected data-quality issues** (so the data-quality layer is real):
40 duplicate clients, 8 negative settlements, 10 negative `days_in_stage`, 6
close-before-open dates, ~3% missing emails. Full column-level detail in
[`documentation/data_dictionary.md`](documentation/data_dictionary.md).

## 🧮 SQL Highlights

[`sql/`](sql/) contains the full analytical SQL, written in PostgreSQL with
advanced techniques throughout:

- **Workup funnel** built with a stage-ordering CTE + `FIRST_VALUE`/`LAG` window
  functions for stage-over-stage conversion.
- **Case-manager productivity** ranked with `RANK() OVER (...)`, blending caseload,
  completion and activity volume.
- **SLA compliance** using per-stage targets joined to open inventory.
- **Inactive-case triage** via `ROW_NUMBER()` partitioned by manager and date
  bucketing with `CASE`.
- **Monthly intake trends** with `LAG` MoM growth and a rolling 3-month average
  window frame.
- **Data quality**: duplicate detection on an identity key, range/logical
  violation checks, NULL audits and referential-integrity (orphan) checks.

Files: `01_create_tables.sql`, `02_load_data.sql`, `03_kpi_queries.sql`,
`04_dashboard_queries.sql`, `05_data_quality_queries.sql`.

## 🤖 Machine Learning & Forecasting Summary

### Predictive model — *Will this matter complete workup?*
[`python/04_predictive_model.py`](python/04_predictive_model.py)

| Model | Accuracy | Precision | Recall | ROC-AUC (test) | ROC-AUC (5-fold CV) |
|-------|:-------:|:--------:|:------:|:-------:|:-------:|
| **Logistic Regression** (selected) | 0.687 | 0.701 | **0.844** | 0.696 | **0.700 ± 0.020** |
| Random Forest | 0.644 | 0.715 | 0.691 | 0.686 | 0.696 ± 0.014 |

- **Top drivers:** `missing_documents`, `medical_records_received`,
  `communication_count`, attorney experience — all *operationally actionable*.
- **184 of 686** held-out matters flagged **AT RISK**; exported to
  [`python/outputs/predictions.csv`](python/outputs/predictions.csv) for triage.
- **Leakage discipline (and an honest correction):** fields that encode the
  outcome (`current_stage`, `case_status`, `case_outcome`, `settlement_amount`,
  `date_closed`, `workup_completed_date`) are excluded. An adversarial
  methodology review then caught that **`days_in_stage` was also a leak** (it is
  derived from the funnel stage, which is conditional on the target). I removed
  it and re-reported — ROC-AUC fell from ~0.715 to **0.696**, and *that lower
  number is the credible one*. Imputation runs inside the pipeline (fit on train
  folds only) and metrics are cross-validated. See
  [`documentation/methodology_and_limitations.md`](documentation/methodology_and_limitations.md).

### Forecast — *Monthly workup demand (new matters entering the pipeline)*
[`python/03_forecasting.py`](python/03_forecasting.py)

- Regularized regression with explicit trend + monthly seasonality.
- **Target choice:** forecasts *intake/opened* volume, not *completion* volume.
  Completion volume is right-censored (recent matters haven't finished workup),
  which corrupts a back-test; intake is the uncensored leading indicator the
  workup team plans capacity against.
- **Back-test MAPE 12.5%** (MAE ≈ 15.8 matters/month; last 4 months held out).
  Interval is anchored on back-test error and widens with horizon.
- Next 6 months ≈ **162, 157, 147, 154, 122, 127** new matters/month.

Figures: `roc_curve.png`, `feature_importance.png`, `forecast_workup_volume.png`,
and five EDA charts in [`python/outputs/`](python/outputs/).

## 💡 Business Recommendations (headline)

Full, finding-linked detail in
[`documentation/executive_recommendations.md`](documentation/executive_recommendations.md).

1. **Attack the records bottleneck.** The Records Requested/Received stages are
   the choke point; a dedicated records-chasing pod and SLA enforcement is the
   single highest-leverage move on cycle time.
2. **Fix the mass-tort workup gap.** Mass-tort dockets complete workup at
   ~41–51% vs. 73–75% for core PI. Standardize a mass-tort workup playbook.
3. **Operationalize the risk model.** Feed the **AT-RISK** list into a weekly
   triage so stalling matters are worked *before* they age out.
4. **Manage to capacity, not headcount.** Use the forecast + caseload-adjusted
   productivity to balance assignments against each manager's monthly capacity.
5. **Institutionalize data quality.** Enforce dedup at intake and validation on
   settlement/date fields to keep every downstream KPI trustworthy.

## ▶️ How to Run the Project

```bash
# 1) From the project root, create a virtual environment & install deps
python3 -m venv .venv
source .venv/bin/activate              # Windows: .venv\Scripts\activate
pip install -r python/requirements.txt

# 2) Generate the synthetic data (writes data/*.csv)
python python/generate_data.py

# 3) Run the analytics pipeline in order
python python/01_data_cleaning.py      # -> data/processed/*.csv + cleaning report
python python/02_eda.py                # -> python/outputs/eda_*.png
python python/03_forecasting.py        # -> forecast csv + chart
python python/04_predictive_model.py   # -> metrics, ROC, importances, predictions.csv
```

**Load into a database (optional, for the SQL & BI layers):**
```bash
createdb legal_intel
psql -d legal_intel -f sql/01_create_tables.sql
psql -d legal_intel -f sql/02_load_data.sql        # run from project root (uses \copy data/*.csv)
psql -d legal_intel -f sql/03_kpi_queries.sql      # explore KPIs
```

Then point Power BI / Sigma at the database (or the CSVs) and build the seven
pages per [`dashboards/dashboard_specifications.md`](dashboards/dashboard_specifications.md).
